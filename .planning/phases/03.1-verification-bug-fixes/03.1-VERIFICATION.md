### Human Verification Required

The following items cannot be confirmed programmatically — they require a running Convex backend and browser:

#### 1. Dev Mailbox Email Capture

**Test:** Start `npm run dev`. Trigger OTP sign-in with any email. Visit `/dev/mailbox`.
**Expected:** The OTP email appears in the mailbox list with subject "Sign in to Feather Starter" and the token visible.
**Why human:** `storeDevEmail` in `ResendOTP.ts` calls `new ConvexHttpClient(process.env.CONVEX_URL)`. If `CONVEX_URL` is not set in the Convex action runtime context, it silently returns without storing. This is the open uncertainty from PLAN-CHECK Warning 3 — cannot confirm without a live run.

#### 2. Live Auth Error Messages

**Test:** On the login page, sign up with an email that already has an account.
**Expected:** "An account with this email already exists. Try signing in." appears below the submit button.
**Why human:** The catch block matches on `msg.includes("already exists")` — the actual Convex auth error string must contain that substring for the specific message to show. Tests mock the signIn function; live auth needed.

#### 3. Provider Hiding in Production-like Setup

**Test:** Start dev with no `AUTH_RESEND_KEY` or `AUTH_GITHUB_ID` in `.env.local`. Visit `/login`.
**Expected:** "Or continue with" divider and all OTP/GitHub buttons are completely absent from the page.
**Why human:** The `availableProviders` query runs against the live Convex backend. Browser visual confirmation needed.

#### 4. Login Layout Visual Check

**Test:** Visit `/login`.
**Expected:** Form is centered on the page. No left panel with quotes. Logo at top-left. Background is card color.
**Why human:** CSS class `flex min-h-screen items-center justify-center` is present but rendering depends on browser and Tailwind compilation.

#### 5. Email Persistence Across Auth Form Switches

**Test:** On `/login`, type an email into the password form. Click "Forgot password?". Check the forgot-password form's email field.
**Expected:** The email you typed appears pre-filled in the forgot-password form's email input.
**Why human:** State-lifting wiring is verified in tests (the `onEmailChange` callback fires), but the full user flow across form transitions needs live browser confirmation.

#### 6. Settings Form Re-sync

**Test:** In settings, note current username. Open browser devtools, manually patch the `users` table via Convex dashboard to change username. Observe settings form.
**Expected:** Form input updates to reflect the new server value without page refresh.
**Why human:** Requires live Convex real-time subscription to trigger and be observable.
