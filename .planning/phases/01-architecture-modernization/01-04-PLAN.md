**Step 1: Split translation files**

Read the current `public/locales/en/translation.json` and `public/locales/es/translation.json`. Split each into namespace files:

- `common.json` -- shared strings (app name, generic buttons, generic labels)
- `auth.json` -- login, email, OTP strings
- `dashboard.json` -- dashboard page strings
- `settings.json` -- settings page strings (general tab)
- `billing.json` -- billing/checkout/subscription strings
- `onboarding.json` -- username setup strings

Each namespace file contains only the keys relevant to that feature. If a key is used across multiple features, put it in `common.json`.

Do this for BOTH `en/` and `es/` locales.

**Step 2: Update i18n.ts**

Update `src/i18n.ts` to use namespace-based loading:
- Add `ns: ["common", "auth", "dashboard", "settings", "billing", "onboarding"]`
- Set `defaultNS: "common"`
- Update the backend `loadPath` to use `{{ns}}`: `"/locales/{{lng}}/{{ns}}.json"`
- Remove the old `translation` namespace reference if present

**Step 3: Update useTranslation calls**

Search the codebase for `useTranslation()` calls. Update each to specify the correct namespace:
- Dashboard components: `useTranslation("dashboard")`
- Settings components: `useTranslation("settings")`
- Billing components: `useTranslation("billing")`
- Onboarding components: `useTranslation("onboarding")`
- Auth components: `useTranslation("auth")`
- Shared/common: `useTranslation("common")` or `useTranslation()` (uses defaultNS)

Keys may need to be adjusted if the old keys had feature prefixes that are now redundant (e.g., `settings.title` in settings namespace becomes just `title`).

**Step 4: Remove old translation.json files**

Delete `public/locales/en/translation.json` and `public/locales/es/translation.json` after splitting.

**Step 5: Verify**

All translated strings must still render correctly. Run tests and typecheck.
